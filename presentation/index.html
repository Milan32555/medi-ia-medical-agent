PHASE 1 — ORIGINAL PROPOSAL
What We Proposed in Phase 1
🧬

NLP Engine

BioBERT / BETO — pre-trained biomedical transformers for dense Spanish-language symptom embeddings. Designed for clinical text comprehension.

📚

Knowledge Base

SNOMED CT ontology — international clinical terminology standard with thousands of structured medical concepts in Spanish.

✍️

Generation

FLAN-T5 — generative transformer model for producing natural language diagnostic summaries from retrieved context.

⚡

Vector Store

FAISS Dense (float32) — dense vector index over SNOMED CT entries for semantic similarity retrieval.

🖥️

Frontend

Vue 3 SPA consuming a Flask REST API. Modern reactive interface with component-based architecture.

🏥

Triage

3-level system: leve / moderada / grave. Cloud/GPU deployment on Google Colab Pro or HuggingFace Endpoints.

⚠ PHASE 1 DEPENDENCY RISKS (identified before implementation)
BioBERT + FLAN-T5 require 4–6 GB GPU VRAM · SNOMED CT in Spanish requires institutional licensing · Vue 3 adds frontend framework complexity
JUSTIFIED PIVOT
Why the Proposal Changed — 3 Converging Reasons
💻

1. Computational Constraints

BioBERT and FLAN-T5 require a minimum of 4–6 GB GPU VRAM for inference. No GPU was available in the local development environment. CPU-only inference was prohibitively slow.

📄

2. Corpus Feasibility

Obtaining and processing SNOMED CT in Spanish required institutional licensing and significant preprocessing effort beyond the scope and timeline of this phase.

👩‍🏫

3. Instructor Guidance

Prof. Alexandra La Cruz recommended a HuggingFace + Sentence Transformers approach — embedding a local medical knowledge corpus. This preserves RAG while enabling CPU deployment.

✅ THE CORE RAG INSIGHT REMAINS IDENTICAL
"Retrieve relevant context, then generate a grounded response." The architectural principle from Phase 1 is preserved. Only the implementation layer changed: dense transformer embeddings → sentence-transformers + TF-IDF, and FLAN-T5 → Qwen2.5-7B via HuggingFace free API.

REACT PATTERN — Yao et al. (2023)
How the ReAct Agent Thinks and Acts
THOUGHT 1

"The user reports high fever, sudden onset, muscle pain, and chills. I need to search the indexed medical books for relevant clinical context before answering."

ACTION

Calls tool: search_symptoms("high fever sudden onset muscle pain rigors chills")

OBSERVATION

Retrieved: Harrison p.136 — "Fever, rigors, myalgia..." · Oxford Handbook p.831 — "Symptoms: Fever, malaise, dyspnoea, purulent sputum..." · Symptom to Diagnosis p.87 — "Fever with chills suggests bacteremia..."

ACTION

Calls tool: assess_urgency("high fever sudden muscle pain chills|||Harrison p.136 Oxford p.831...")

OBSERVATION

Result: emergencia — pattern matches severe infectious disease / bacteremia indicators per clinical literature.

FINAL ANSWER

Structured diagnosis: condition, urgency level, clinical reasoning with citations from Harrison + Oxford Handbook, recommendation. Disclaimer appended.

AUTONOMOUS AGENT TOOLS
4 Medical Tools the Agent Decides to Use

⚙ search_symptoms(query)

Primary retrieval tool. FAISS similarity search → cross-encoder reranker → top-4 chunks with book + section + page. Always called first for any symptom query.

⚙ assess_urgency(symptoms, context)

Evaluates triage level: leve / moderada / grave / emergencia 🚨. Uses clinical keywords + retrieved context. Triggers emergency banner in UI.

⚙ get_drug_info(condition)

Retrieves medication information from The Top 100 Drugs Clinical. Called when the agent identifies a condition requiring pharmacological context or treatment options.

⚙ get_section(book, topic)

Deep-dives into a specific book + section. Used when the agent needs more detail from a particular reference. Input: Harrison|||chest pain or Oxford|||respiratory.

🧠 WHAT MAKES THIS A REAL AGENT
The LLM autonomously decides which tools to call, in what order, and when it has enough context to generate a final answer. It is not a fixed pipeline — it reasons. The full trajectory (Thought → Action → Observation) is displayed in the UI for explainability. (XAI principle)
MODELS & TECHNOLOGY STACK
All Models Used in Phase 2 — And Why
🤖

Qwen/Qwen2.5-7B-Instruct

Role: Main LLM for reasoning, tool selection, and response generation.
Why: Multilingual (ES + EN), free via HuggingFace Inference API, strong instruction-following. Verified working — only model available free in HF chat API at time of integration.

🔢

paraphrase-multilingual-MiniLM-L12-v2

Role: Semantic embedding of book chunks + user queries.
Why: Multilingual (50+ languages including ES), 384-dim vectors, ~400MB, runs on CPU. Recommended by Professor La Cruz. Enables Spanish queries over English book content.

🎯

cross-encoder/ms-marco-MiniLM-L-6-v2

Role: Reranker — scores (query, chunk) pairs for true relevance.
Why: Cross-encoders see both query and passage jointly → dramatically more accurate than cosine similarity alone. Critical for medical precision.

⚡

FAISS-CPU + TF-IDF (dual mode)

Role: Vector database for similarity search.
Why: FAISS = semantic retrieval from real books (post-ingest). TF-IDF = lightweight fallback when index not present. Justification for TF-IDF: CPU-only, XAI-interpretable weights, effective for 20-condition corpus.
